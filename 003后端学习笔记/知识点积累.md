## 1. QueryWrapper vs XML 手写 SQL —— 什么场景用哪种

**核心结论：能用 QueryWrapper 就用 QueryWrapper（少写代码），QueryWrapper 搞不定的就写 XML（灵活度高）。**

| 场景 | 用哪种 | 原因 |
|------|--------|------|
| 按主键查/改/删 | QueryWrapper / 内置方法 | `selectById(1)` 一行搞定 |
| 单表 + 简单条件 | QueryWrapper | `.eq().gt().orderByDesc()` 很方便 |
| 单表 + 动态条件多 | QueryWrapper | Java 里 if 判断比 XML 的 `<if>` 直观 |
| 多表 JOIN | XML 手写 | QueryWrapper 不支持 JOIN |
| GROUP BY + HAVING | XML 手写 | 聚合统计用 SQL 写更清晰 |
| 子查询 / CTE | XML 手写 | 复杂逻辑必须手写 SQL |
| 需要精确控制返回列 | XML 手写 | 可以精确写 SELECT 哪些列 |

**关系图：**

```
Mapper 接口 extends BaseMapper
├── 继承的方法（自动生成 SQL）→ 配合 QueryWrapper 使用
│   selectById / selectList / insert / updateById / deleteById
│
└── 自己声明的方法（SQL 写在 XML 里）→ 处理复杂查询
    selectNotHasRelation... / selectClientsWithMultiple... 等
```

## 2. @Resource vs @Autowired —— 依赖注入的两种方式

| 特性 | `@Resource` | `@Autowired` |
|------|------------|--------------|
| 来源 | JSR-250（Java 标准） | Spring 专有 |
| 默认匹配方式 | 按名称（byName） | 按类型（byType） |
| 找不到名称时 | 退回按类型匹配 | 直接按类型匹配 |

**项目中的写法**：大部分用 `@Resource`，建议统一使用 `@Resource`。注入时应尽量注入**接口类型**而非具体实现类。

## 3. @RestController = @Controller + @ResponseBody

- `@Controller`：声明这是一个 MVC 控制器
- `@ResponseBody`：返回值直接写入 HTTP 响应体（JSON），不走 ViewResolver
- `@RestController`：两者合一

本项目所有 ServiceImpl 都用 `@RestController`，所以返回值自动通过 Jackson 序列化为 JSON，不需要额外配置。

## 4. Spring MVC 请求处理全流程

```
请求进入
  │
  ▼ DispatcherServlet（前端控制器，spring-boot-starter-web 自动配置）
  │
  ▼ HandlerMapping（根据 @RequestMapping 找到对应方法）
  │
  ▼ HandlerAdapter（解析 @RequestParam 等参数，调用方法）
  │
  ▼ Controller 方法执行（IoC 容器通过 DI 注入了所有依赖）
  │
  ▼ HttpMessageConverter（Jackson 把返回值序列化为 JSON）
  │
  ▼ 返回 HTTP 响应
```

## 5. Spring Cloud 微服务启动流程

```
项目启动
  │
  ▼ 读 bootstrap.yml（优先级最高）
  │
  ▼ 从 Eureka 查找 Config Server 地址
  │
  ▼ 从 Config Server 拉取完整配置（数据库、Redis、ES 等地址）
  │
  ▼ 初始化 Spring 上下文（创建所有 Bean、建立数据库连接池等）
  │
  ▼ Liquibase 自动检查并执行未跑过的 SQL 变更
  │
  ▼ 向 Eureka 注册自己（服务名 + IP + 端口）
  │
  ▼ 启动完成，开始接收请求
```

## 6. 项目中的数据存储分工

| 存储 | 存什么 | 为什么 |
|------|--------|--------|
| PostgreSQL | 主业务数据（客户、商机、带看等） | 关系型数据，事务保证 |
| Oracle | 老系统数据（通过 `@DS` 切换） | 历史遗留，逐步迁移 |
| Redis | 热数据缓存（先查缓存再查库） | 读多写少场景提升性能 |
| Elasticsearch | 搜索/列表/聚合 | 支持复杂搜索、模糊匹配 |
| MongoDB | 跟进记录（CustTrack） | 文档结构灵活，字段不固定 |
| FastDFS | 文件/图片 | 分布式文件存储 |

## 7. SQL 执行顺序 vs 书写顺序

书写顺序：

```sql
SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
```

实际执行顺序：

```
① FROM      先确定从哪张表
② WHERE     过滤行
③ GROUP BY  分组
④ HAVING    过滤组
⑤ SELECT    确定返回哪些列
⑥ ORDER BY  排序
⑦ LIMIT     截取条数
```

## 8. bootstrap.yml vs application.yml

| | bootstrap.yml | application.yml |
|--|---------------|-----------------|
| 加载时机 | 最先加载 | 之后加载 |
| 用途 | 配置"配置中心"本身的信息 | 配置应用自身的信息 |
| 典型内容 | Eureka 地址、Config Server 名称 | 数据库、Redis、业务配置 |

本项目只有 `bootstrap.yml`，因为大部分配置从 Config Server 远程拉取。

## 9. Feign 调用流程（微服务间通信）

```
服务 A 调用服务 B：

服务 A 的代码：
@FeignClient(value = "landz-client-domain", path = "/client-domain/client-biz-opportunity")
public interface IClientBizOpportunityService {
    @PostMapping("/getClientBizOpportunityIdByCustId")
    Long getClientBizOpportunityIdByCustId(@RequestParam("custId") Long custId);
}

执行流程：
服务 A 调用接口方法
  │
  ▼ Feign 拦截，从 Eureka 查 "landz-client-domain" 的实际地址
  │
  ▼ 发起 HTTP 请求：POST http://实际IP:8509/client-domain/client-biz-opportunity/getClientBizOpportunityIdByCustId?custId=123
  │
  ▼ 服务 B（本项目）的 DispatcherServlet 接收并处理
  │
  ▼ 返回结果，Feign 反序列化为 Long
```

# Codex 使用中文手册

## 1. 这份文档适合谁

这份文档适合已经开始使用 Codex，或者准备把 Codex 用到日常开发中的人。

重点不是讲概念，而是讲：

- Codex 是什么
- App、CLI 分别怎么用
- 常用能力有哪些
- 模型怎么选
- 权限、沙箱、worktree 是什么
- 日常开发怎么和 Git、IDE、终端配合

本文基于 OpenAI 官方 Codex 文档整理，时间为 `2026-04-29`。

---

## 2. Codex 是什么

Codex 可以理解成 OpenAI 提供的“编码代理（coding agent）”。

它不只是回答代码问题，还可以在你授权的范围内：

- 阅读项目代码
- 修改文件
- 运行终端命令
- 调用浏览器、MCP、插件等工具
- 帮你做代码审查、调试、文档整理、自动化任务

官方对 Codex app 的描述是：它是一个专注的桌面端工作区，用来并行处理 Codex 线程，并且内置了 `worktree`、`automations` 和 `Git` 功能。  
来源：[Codex app](https://developers.openai.com/codex/app)

---

## 3. Codex 有哪几种使用方式

官方文档把 Codex 分成几个主要入口：

- Codex App：桌面应用，适合日常主工作流
- Codex CLI：终端里直接使用
- IDE Extension：在 IDE 中配合使用
- Web：网页环境

对大多数开发者来说，最常见的是：

1. `Codex App`
2. `Codex CLI`

如果你已经习惯在终端里工作，CLI 会非常顺手。  
如果你想更方便地管理线程、项目、review、自动化、worktree，App 更完整。

---

## 4. Codex App 适合做什么

官方说明里，Codex App 的核心特点有几个：

- 支持并行处理多个线程
- 支持多个项目切换
- 内置 Git、worktree、automations
- 可以配合本地终端、浏览器、代码审查界面使用

官方文档提到，Codex App 可以在一个窗口里跨多个项目工作；每个项目本质上都像是在某个目录里启动一次 Codex 会话。  
来源：[Codex app features](https://developers.openai.com/codex/app/features)

### 典型使用方式

- 一个项目对应一个代码仓库
- 一个线程对应一个具体任务
- 复杂任务用单独线程，不要都混在一个线程里
- 需要后台跑的任务交给 worktree 或 automation

## 5. Codex CLI 怎么安装和启动

官方文档给出的 CLI 安装方式：

```bash
npm i -g @openai/codex
```

启动：

```bash
codex
```

首次运行时，Codex 会提示你登录，可以使用：

- ChatGPT 账号
- 或 API Key

官方说明：CLI 可以在你当前目录中读取、修改、运行代码。  
来源：[Codex CLI](https://developers.openai.com/codex/cli)

### 升级 CLI

```bash
npm i -g @openai/codex@latest
```

## 6. 日常最常见的使用方式

### 方式 1：在项目目录中启动 Codex

```bash
cd /你的项目目录
codex
```

例如：

```bash
cd /Users/zzy/AndroidStudioProjects/mynote/mynote
codex
```

这样 Codex 会把当前目录作为主要工作区。

### 方式 2：指定目录启动

```bash
codex -C /你的项目目录
```

适合你当前终端不在目标目录里，但想直接进入某个项目。

### 方式 3：让 Codex 处理具体任务

例如你可以直接输入自然语言：

- “帮我分析这个崩溃日志可能在哪一层出问题”
- “把登录页相关代码找出来，并说明请求链路”
- “修复这个编译错误，只改必要文件”
- “写一份中文 README，放到 docs 目录里”

## 7. Codex 最常用的能力

### 1. 读代码

适合：

- 看某个功能入口
- 查某个字段在哪里赋值
- 分析类之间的调用关系
- 找 bug 来源

### 2. 改代码

适合：

- 修 bug
- 小范围重构
- 批量替换
- 补注释、补文档、补测试

### 3. 跑命令

适合：

- 编译
- 运行测试
- 查看日志
- 查 Git 状态
- 搜索项目文件

### 4. 做 review

官方文档说明，使用 `/review` 后，代码审查结果会直接显示在 review 面板里。  
来源：[Codex app review](https://developers.openai.com/codex/app/review)

### 5. 用工具

Codex 可以接入：

- MCP 服务器
- 浏览器
- Skills
- 插件
- 本地终端

这使它不只是“聊天模型”，而更像一个可执行任务的工程助手。

## 8. Slash Commands 常用命令

官方文档列出了 Codex App 里的常用斜杠命令。

### App 中常见命令

- `/status`：查看线程 ID、上下文占用、限额等状态
- `/review`：进入代码审查模式
- `/mcp`：查看 MCP 状态
- `/plan-mode`：切换到多步骤计划模式
- `/feedback`：提交反馈

来源：[Codex app commands](https://developers.openai.com/codex/app/commands)

### CLI 中常见命令

- `/model`：切换当前模型
- `/permissions`：切换权限/审批模式
- `/status`：查看当前状态

官方文档还提到，CLI 里可以用 `/personality` 切换沟通风格。  
来源：[Codex CLI slash commands](https://developers.openai.com/codex/cli/slash-commands)

## 9. 模型怎么选

根据 OpenAI 官方 Codex 模型文档，当前推荐是：

- 优先使用 `gpt-5.5`
- 如果 `gpt-5.5` 还不可用，则使用 `gpt-5.4`
- 轻量任务或 subagents 可以用 `gpt-5.4-mini`

官方原意是：

- `gpt-5.5`：适合复杂编码、computer use、知识型工作流、研究类任务
- `gpt-5.4`：次一档但仍然很强
- `gpt-5.4-mini`：更快、更省，适合轻任务
- `gpt-5.3-codex`：复杂软件工程的专门编码模型
- `gpt-5.2`：上一代通用编码/代理模型

来源：[Codex models](https://developers.openai.com/codex/models)

### 简单选型建议

- 大多数正式开发任务：`GPT-5.5`
- 日常改 bug、读代码：`GPT-5.4`
- 快速小任务：`GPT-5.4-Mini`
- 明确偏 agentic coding：`GPT-5.3-Codex`

## 10. 权限和沙箱要怎么理解

这是 Codex 很重要的一部分。

官方文档强调 Codex 有两个关键控制项：

- `Approval mode`：什么情况下必须先问你
- `Sandbox mode`：它能读写哪些目录、能不能执行更高风险操作

官方建议：如果你刚开始使用 coding agent，先使用默认权限，不要一上来就把权限放太大。  
来源：[Best practices](https://developers.openai.com/codex/learn/best-practices)

### 你可以这样理解

- 权限保守：更安全，但很多操作会频繁询问
- 权限放宽：更高效，但风险更高

### 官方建议的思路

- 新仓库、新工作流，先保守
- 熟悉后，再逐步放开可信目录或可信命令
- 不要一开始就给全盘高权限

### CLI 中调整权限

可以用：

```text
/permissions
```

来切换审批模式。  
来源：[Codex CLI slash commands](https://developers.openai.com/codex/cli/slash-commands)

## 11. worktree 是什么，为什么 Codex 很强调它

官方文档对 `worktree` 支持非常完整，因为它很适合代理式开发。

### worktree 的实际价值

你可以理解为：

- 同一个 Git 仓库
- 同时展开多个工作目录
- 每个目录可以服务不同线程或不同分支
- 避免不同任务互相污染

官方文档说明：

- Codex 可以把线程放到 worktree 中运行
- 线程可以在 `Local` 和 `Worktree` 之间安全切换
- Codex 会处理其中涉及的 Git 步骤
- Codex 管理的 worktree 默认在 `$CODEX_HOME/worktrees`

来源：[Codex app worktrees](https://developers.openai.com/codex/app/worktrees)

### 为什么这很重要

如果你：

- 本地正在开发 A 需求
- 又想让 Codex 后台处理 B 任务

那直接混在同一个目录里，风险很高。  
而 worktree 的意义就是把任务隔离开。

官方最佳实践里也明确提到：不要在同一批文件上同时开多个活动线程，而不使用 Git worktrees。  
来源：[Best practices](https://developers.openai.com/codex/learn/best-practices)

## 12. Local 和 Worktree 怎么选

### 用 Local

适合：

- 你就在主工作目录里开发
- 你希望 IDE、终端、运行环境都和现在一致
- 你需要立即手动验证代码

### 用 Worktree

适合：

- 希望把 Codex 的修改和你当前本地工作隔离
- 想让它后台独立处理任务
- 不想污染当前分支或当前工作区

官方文档说明，线程可以在 Local 和 Worktree 之间 handoff，Codex 会处理切换需要的 Git 动作。  
来源：[Codex app worktrees](https://developers.openai.com/codex/app/worktrees)

## 13. Automations 是什么

Automations 可以理解成“定时让 Codex 自动做某件事”。

官方文档提到：

- 自动化任务可以跑在本地项目，也可以跑在新的 worktree 中
- 跑在 worktree 更适合隔离自动化修改
- 自动化会复用你的默认 sandbox 设置

来源：[Codex app automations](https://developers.openai.com/codex/app/automations)

### 适合做什么

- 定时检查某个任务是否完成
- 定时做代码扫描或 review
- 定时检查 GitHub / Slack / PR 状态
- 提醒你继续某个线程

### 使用建议

官方建议也很务实：

- 先把任务手动跑顺
- 再变成 automation
- 不要把不稳定流程直接自动化

## 14. AGENTS.md 是什么

`AGENTS.md` 可以理解成“给 Codex 的长期工作说明书”。

官方最佳实践提到：

- 可以有全局的 `~/.codex/AGENTS.md`
- 也可以有仓库级 `AGENTS.md`
- 还可以在子目录放更具体的 `AGENTS.md`
- 离当前目录越近的规则，优先级越高

来源：[Best practices](https://developers.openai.com/codex/learn/best-practices)

### 适合写什么

- 项目编码规范
- 提交规范
- 测试命令
- 哪些目录不能动
- 哪些模块修改前必须先看什么文档

### 官方建议

`AGENTS.md` 要短、准、实用。  
不要写一大堆空泛原则。

## 15. MCP、Skills、插件分别是什么

### MCP

MCP 是 Codex 连接外部工具和数据源的标准方式。

例如：

- 接内部平台
- 接知识库
- 接设计工具
- 接 GitHub、任务系统、数据库

官方设置页说明，MCP 配置也会影响 CLI 和 IDE，因为配置写在 `config.toml` 中。  
来源：[Codex app settings](https://developers.openai.com/codex/app/settings)

### Skills

Skill 是一套本地说明，用来告诉 Codex：

- 在什么场景触发
- 应该怎么做
- 用哪些工具

你可以把它理解成“可复用工作流模板”。

### 插件

插件通常是更成体系的能力集合，可能包含：

- Skills
- MCP
- 额外指令
- 应用能力

## 16. 浏览器能力怎么用

如果你在做前端、后台管理系统、H5、文档站，这个能力很有用。

官方文档里给出的典型例子是，让浏览器打开 `http://localhost:3000/settings`，复现页面布局问题，再只修复溢出的控件。  
来源：[Codex app in-app browser](https://developers.openai.com/codex/app/browser)

### 适合场景

- 打开本地前端页面
- 复现 UI 问题
- 检查交互流程
- 联动代码修改和页面验证

官方设置页里，最值得关注的是：

- 文件如何打开
- 命令输出展示多少
- 多行输入行为
- 线程运行时是否阻止系统休眠

## 18. 实际开发中的推荐工作流

### 场景 1：读代码和分析问题

做法：

1. 在项目目录启动 Codex
2. 直接描述你要分析的问题
3. 让它先定位相关文件、入口、调用链
4. 如果问题复杂，让它先出分析结论，再决定是否改代码

### 场景 2：修一个明确 bug

做法：

1. 说明 bug 现象
2. 指定只改必要文件
3. 要求先复现、再修复、再验证
4. 最后让它总结改动点

### 场景 3：你本地继续开发，同时让 Codex 处理另一个任务

做法：

1. 当前本地目录保留不动
2. 让 Codex 在 worktree 里处理任务
3. 需要时再 handoff 回 Local 或创建分支

### 场景 4：例行检查任务

做法：

1. 先手动把流程跑通
2. 再建立 automation
3. 优先跑在 worktree 中，避免污染主目录

## 19. 提示词怎么写更有效

官方最佳实践虽然没有鼓励“花哨提示词”，但强调了几个方向：

- 多步骤复杂任务要先规划
- 让 Codex 能看到如何构建和测试
- 不要把太多长期规则都塞进单次 prompt，应该放到 `AGENTS.md`

来源：[Best practices](https://developers.openai.com/codex/learn/best-practices)

### 推荐写法

不要只说：

```text
帮我修一下
```

更推荐：

```text
请先定位登录页面提交按钮点击无响应的原因；
只修改必要文件；
修复后说明根因、改动点、验证方式；
如果需要运行命令，先告诉我你要做什么。
```

### 再复杂一点的写法

```text
请先阅读支付模块相关代码，梳理从页面点击到接口请求再到结果回显的完整链路；
先不要改代码；
先给出问题最可能出现的 3 个位置，并说明判断依据。
```

## 20. 新手最容易踩的坑

### 1. 一上来就给太大权限

不建议。

先保守，再放开。

### 2. 在同一目录里并行跑多个互相影响的任务

容易把工作区弄乱。  
优先用 `worktree` 隔离。

### 3. 任务描述太模糊

会导致 Codex 花很多步数在猜你的意图。

### 4. 没告诉它怎么验证

如果你知道项目怎么编译、怎么跑测试、怎么打开页面，最好明确说出来。

### 5. 把临时要求反复手写

适合沉淀进 `AGENTS.md` 或 Skill。

## 23. 最后建议

如果你是刚开始用 Codex，最实用的路线不是先研究全部功能，而是按下面顺序上手：

1. 先学会在项目目录里启动 Codex
2. 先用它读代码、查问题
3. 再让它做小范围改动
4. 再学 `review`、`worktree`
5. 最后再上 `automation`、`MCP`、`skills`

这样上手最快，也最不容易把环境和流程搞乱。
